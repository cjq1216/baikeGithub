    1. **改 methods**: `@user.route('/search', methods=['GET'])`(从 POST 改 GET,D-45)

    2. **改入参**: `q = request.args.get('q', '').strip()`(从 `request.form.get` 改 `request.args.get`,空字符串兜底,D-45)

    3. **改返回逻辑**:
       ```python
       @user.route('/search', methods=['GET'])
       def search():
           q = request.args.get('q', '').strip()
           lemmas = []
           if q:
               lemmas = Lemma.query.filter(Lemma.title.like('%' + q + '%')).limit(20).all()
           # HTMX 请求:直接返回片段;非 HTMX:返完整 result.html(走浏览器 GET 渐进增强)
           if request.headers.get('HX-Request'):
               return render_template('_search_result.html', lemmas=lemmas)
           if not lemmas and q:
               flash('所查词条不存在，工作人员正在努力完整词条库～～')
               return redirect(url_for('apple.home'))
           if not lemmas:
               return render_template('result.html', results=[])
           return render_template('result.html', results=lemmas)
       ```

    关键点:
    - **D-65 配套**:`/user/detail` 同步改 GET `?title=...`(Plan 4.3 T2 实施),result.html 链接 `url_for('apple.detail', title=...)` 自动生成 `?title=foo`
    - 20 条上限:D-45 决策(`limit(20)`)

3. **T3: Path traversal / SQL injection via `?q=` query**
   - Vector: 用户在搜索框输入 `' OR 1=1 --`,`/user/search` 拼接 SQL 注入。
   - Mitigation: **D-65 + Phase 1**:使用 `Lemma.query.filter(Lemma.title.like('%' + q + '%'))` — SQLAlchemy `like()` 是参数化查询,`q` 走 `?` 占位符绑定,**不**做字符串拼接。`'%' + q + '%'` 是 Python 字符串拼,SQLAlchemy ORM 内部转义。**不**用 `db.session.execute(text('SELECT * FROM lemma WHERE title LIKE %q%'))` 这种 raw SQL 模式(没有这条代码,只是防御性说明)。
   - Test: 实际 SQL `EXPLAIN` 看 `WHERE title LIKE '%<input>%'` 参数绑定,而非字符串拼接。

4. **T4: result.html 中 `{{ result.content | safe }}` 残留**
   - Vector: result.html 显示词条内容截断,但用 `|safe` 跳过转义,攻击者构造词条时塞 `<script>`,搜索结果列表里也触发。
   - Mitigation: **本 plan 改写后**,result.html 用 `{{ (result.content|striptags)|truncate(200) }}` — `striptags` 剥所有 HTML 标签(再用 `truncate`),**不**用 `|safe`,XSS 屏蔽。detail.html 的 `{{ fullcon.content | wikilink | safe }}` 是 Plan 4.3 才实施(且 bleach 在写时已过滤,见 Plan 4.1 T8),`wikilink` filter 内部只识别 `[[...]]` 模式构造安全的 `<a>` 标签(Plan 4.3 实施时 `render_wikilink` 用 `markupsafe.escape` 包裹 title 文本)。
   - Test: 搜索"XSS payload" 词条,result.html 卡片截断文本里 `<script>` 文字被 `striptags` 剥光,只剩 `alert(1)` 文字。

5. **T5: HTMX 注入(被注入 HX-Request header 走片段路径)**
   - Vector: 攻击者让用户浏览器带 `HX-Request: true` 头请求 `/user/search?q=<script>`,HTMX 端点返片段直接插入页面,触发 XSS。
   - Mitigation: **T2 同款 Jinja2 自动转义**,`{{ lemma.title }}` 在片段里也被 escape。`HX-Request` 是普通 HTTP header,任何客户端可设,本身不构成 XSS — XSS 由渲染层 escape 兜底。

6. **T6: 旧搜索 POST 端点残留(若 Plan 4.2 实施不完整)**
   - Vector: `/user/search` 旧 POST 没拆,新 GET 端点加上去后,**两个**端点共存,旧 POST 走整页(无 HTMX)用户拿不到搜索结果。
   - Mitigation: **verification step 2**:`grep -F 'methods=[' app/route/user.py` 中 search 路由只有 `methods=['GET']` 一处,无 `methods=['POST']` 残留。`app/route/user.py:search` **不**含 `request.form.get('searchtext')` 旧入参。

7. **T7: 错误页 traceback 泄露(Phase 2 D-18 兜底)**
   - Vector: 500 错误时,`error.html` 模板如果误用 `e.__traceback__` 暴露 stack trace,泄露代码细节。
   - Mitigation: **本 plan error.html 只用 `error.code` / `error.name` / `error.description`**(Flask `HTTPException` 标准属性,服务器侧构造,无敏感信息)。**不**引用 `e.traceback` / `e.__traceback__` / `error.traceback`。Phase 2 `app.debug=False` 时 Flask 默认不渲染 traceback(只写 stderr),双保险。

Risk: T1 + T2 必须严格执行(CSRF + XSS);T3 / T6 是 SQL 安全 / 端点治理;其余是 UX / 渐进增强相关。
