<objective>
Enable global CSRF protection on every state-changing POST, register a unified Jinja error page for 400/403/404/500, and gate the `/api/reset` development route behind `app.debug` while exposing the same init logic as a `flask init-db` CLI command for production/Phase-5 use.

Purpose: With credentials off-disk (Plan 2.1) and an `is_admin` column in place, this plan closes the cross-site request-forgery vector, eliminates the Flask debug traceback leak, and gives operators a non-HTTP way to bootstrap the database in a container.

Output:
- `requirements.txt` adds `Flask-WTF>=1.2,<2.0`
- `app/__init__.py` instantiates `CSRFProtect(app)`, registers 4 errorhandlers, and registers a `flask init-db` CLI command
- `app/api/model.py` exposes a new module-level `init_db()` function (refactored from the existing reset body)
- `app/api/__init__.py` reset route is guarded with `if not current_app.debug: abort(404)` and delegates to `init_db()`
- 5 templates (`register.html` / `signin.html` / `add.html` / `modify.html` / `result.html`) gain a hidden `csrf_token` field
- New self-contained `app/templates/error.html` renders a friendly error page
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Flask POST handler | All state-changing requests must carry a valid CSRF token issued by the same session |
| anonymous user → /api/reset | In non-debug, must be silently 404'd (no information leak about route existence) |
| any request → error handler | Must not leak traceback in production; only the friendly error page |
| container entrypoint → flask init-db | A trusted SSH/docker-exec context; CLI is not exposed over HTTP |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02-01 | Tampering (CSRF) | all 5 POST forms (regist/login/add/modify/search) | mitigate | Insert `<input type="hidden" name="csrf_token" value="{{ csrf_token() }}">` into every form; Flask-WTF validates token-on-submit (D-09, D-10) |
| T-02-02-02 | Repudiation (CSRF failure) | CSRFProtect raises 400 | mitigate | Override `@app.errorhandler(400)` to flash 'session expired, please retry' and redirect to referrer (D-11) — not the default 400 traceback |
| T-02-02-03 | Information Disclosure (stack trace) | unhandled exceptions in production | mitigate | Register `@app.errorhandler(500)` to render error.html with `error.code` and `error.description` only (D-17, D-18); traceback still goes to stderr (Flask default) |
| T-02-02-04 | Elevation (db reset in prod) | GET /api/reset | mitigate | Wrap the route in `if not current_app.debug: abort(404)` (D-14) — returns 404 via the unified error page; no special-case branch reveals the route |
| T-02-02-05 | Elevation (db reset via CLI) | flask init-db | accept | CLI requires shell access; not exposed over HTTP; this is the explicit production entrypoint (D-13, D-16) |
| T-02-02-06 | Information Disclosure (route existence) | /api/reset 404 in non-debug | accept | Route still exists in the URL map; the 404 reveals nothing about the path; acceptable for a hobby project. A v2 hardening would remove the route registration conditionally on `app.debug`. |
| T-02-02-07 | Spoofing (template forgery) | error.html | mitigate | Template mirrors signin.html (Bootstrap 3 + style.css), static contents only, no user-controlled data is interpolated (only `error.code`, `error.name`, `error.description` which are server-controlled) |
| T-02-02-08 | Tampering (env in error path) | errorhandler(500) | accept | error.html cannot reach DB / env; safe |
| T-02-02-09 | Repudiation (CSRF on logout) | /api/logout | accept | /api/logout is GET; Flask-WTF only protects POST/PUT/PATCH/DELETE; explicitly out of scope (logout is intentionally CSRF-able for a demo product — a v2 hardening would move it to POST + token) |
| T-02-02-SC | Tampering | pip install Flask-WTF | mitigate | Verify Flask-WTF package on PyPI before install; pin to >=1.2,<2.0; treat as [ASSUMED] — not [VERIFIED] via separate audit |
</threat_model>
