<objective>
Add a binary admin role with a moderation capability: a new `admin` blueprint exposing a single `POST /api/admin/lemma/<int:lemma_id>/delete` endpoint, a custom `admin_required` decorator that wraps `@login_required` plus an `is_admin` check, a `flask promote-admin <username>` CLI for ops, and a conditional "delete lemma" button on the lemma detail page that only admins see.

Purpose: Phase 2's hard-auth surface needs a moderation escape hatch. Without it, abusive or low-quality lemmas have no removal path other than touching the DB directly. This plan delivers the minimal moderation capability that Phase 3 (comment delete) and Phase 4 (front-end polish) will build on.

Output:
- New `app/api/admin.py` containing: `admin = Blueprint('admin', __name__)`, `admin_required` decorator, and `delete_lemma(lemma_id)` view
- `app/__init__.py` imports and registers the admin blueprint (url_prefix=/api/admin) and adds a `flask promote-admin <username>` CLI command
- `app/templates/detail.html` gets a conditional block wrapped in `{% if current_user.is_authenticated and current_user.is_admin %}` containing a form-POST button to `url_for('admin.delete_lemma', lemma_id=...)` with a hidden `csrf_token`

Note on ROLE-02 traceability: The `requirements` field in this plan's frontmatter intentionally contains only AUTH-06, ROLE-01, ROLE-03. ROLE-02 (admin can delete any comment) is NOT claimed by this plan because the comment system does not exist yet — ROLE-02 is documented in `REQUIREMENTS.md` and `ROADMAP.md` as deferred to Phase 3 (alongside the rest of the comment system). Phase 2 builds the `admin` blueprint + `admin_required` decorator + `User.is_admin` field as scaffolding; the actual `POST /api/admin/comment/<id>/delete` route ships with Phase 3's comment UI per `02-CONTEXT.md` D-20..D-25 + the "Deferred Ideas" section.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anonymous → /api/admin/* | Must be redirected to /user/login by Flask-Login; no information disclosure about route existence |
| regular user (is_admin=False) → /api/admin/* | Must abort(403), render the unified error page; no leakage of admin-only data |
| admin (is_admin=True) → /api/admin/* | Allowed to delete lemmas; the action is logged via flash + commit |
| shell operator → flask promote-admin | Trusted; CLI is the only way to grant admin status short of direct DB writes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Elevation (privilege escalation) | app/api/admin.py delete_lemma | mitigate | `@admin_required` composes `@login_required` + `current_user.is_admin` check; non-admin returns `abort(403)` which routes through the Plan 2.2 errorhandler |
| T-02-03-02 | Spoofing (CSRF on delete) | app/templates/detail.html delete form | mitigate | Hidden `csrf_token` input required by Plan 2.2 CSRFProtect; CSRF failure on this POST renders the same flash-redirect as any other form |
| T-02-03-03 | Information Disclosure (button visible to non-admin) | detail.html admin block | mitigate | `{% if current_user.is_authenticated and current_user.is_admin %}` gates the button; non-admins and anonymous see no admin-only UI affordance (D-22, ROLE-03) |
| T-02-03-04 | Elevation (mass-assign is_admin via HTTP) | app/api/__init__.py /app/api/admin.py | mitigate | No HTTP route sets `is_admin=True`; the only setter is `flask promote-admin` CLI which requires shell access; the User model constructor accepts `is_admin` for the seeder's sake (Plan 2.1's init_db seeds `a/a` as admin) but no request handler passes that arg |
| T-02-03-05 | Tampering (delete other user's lemma) | delete_lemma view | accept | Admins are explicitly permitted to delete any lemma (ROLE-01) — this is the requirement; the action is irreversible (no soft delete) but acceptable for a moderation tool. A v2 audit log would record `who deleted what when`. |
| T-02-03-06 | Repudiation (404 vs 403 enumeration) | admin_required decorator | accept | Anonymous users get a login redirect (302 to /user/login); authenticated non-admins get 403; the difference reveals "this route exists and you are logged in" — acceptable for an internal demo, would be tightened in a v2 production deployment |
| T-02-03-07 | Tampering (404 lemma_id) | delete_lemma lookup | mitigate | `Lemma.query.get(lemma_id)` returns None for missing; handler flashes 'lemma does not exist' and redirects to home (no traceback) |
| T-02-03-08 | Elevation (forged session) | Flask-Login session integrity | mitigate | Plan 2.1's `FLASK_SECRET` is now env-driven with `instance/.flask_secret` fallback; an attacker cannot forge a session without the secret key |
| T-02-03-09 | Tampering (admin_required bypass) | decorator stacking order | mitigate | `@login_required` is on the inner wrapper (NOT on the outer `admin_required` factory), so an unauthenticated user always gets the login redirect first; the `is_admin` check only runs once a user is already authenticated. This stacking matches the PATTERNS.md canonical shape. |
| T-02-03-SC | Tampering | pip install (no new deps in this plan) | mitigate | n/a — Plan 2.3 adds no new pip packages; Flask-Login and the existing dependencies cover everything |
</threat_model>
