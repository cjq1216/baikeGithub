<objective>
Replace the hardcoded MySQL credentials and Flask `secret_key` with env-var-driven configuration, switch user passwords from plaintext to pbkdf2:sha256 hashed storage, and add length validation (6-30 chars) on username and password in the registration route per D-04. This unlocks the rest of Phase 2 by ensuring no credentials leak in source and that login validation compares hashes safely.

Purpose: Auth and infrastructure are prerequisites for CSRF (Plan 2.2) and admin role (Plan 2.3). Get secrets/credentials off disk and stop storing plaintext passwords before adding any new attack surface. D-04 length validation prevents storage of trivially short or oversized input without introducing a strong-password policy (deferred to v2 per D-04 explicit).

Output:
- `app/__init__.py` reads `FLASK_SECRET` (or writes `instance/.flask_secret`) and reads `DB_*` env vars via `sqlalchemy.engine.url.URL.create`
- `app/api/model.py` `User.password` is `String(255)` (no other model changes)
- `app/api/__init__.py` `registBusiness` validates name/password length (6-30 chars) FIRST, then hashes on create; `loginBusiness` uses `check_password_hash` with a unified failure message
- `.dockerignore` (new) keeps `instance/` and venvs out of build context
</objective>
